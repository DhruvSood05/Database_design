# Smart Elevator Control - Database Design

## Design Objective

Relational database design for a multi-building elevator control system with support for request tracking, elevator assignment, ride history, status logs, and maintenance operations.

## Design Scope

- multiple buildings
- multiple elevators per building
- floor-level request capture
- request-to-elevator allocation history
- ride lifecycle storage
- elevator status telemetry storage
- maintenance workflow storage

## ER Entities

### 1. buildings

Top-level container representing each managed property/facility.

- `building_id` serial PK
- `company_name` varchar(100) not null
- `building_no` int
- `address` text
- `created_at` timestamp
- `updated_at` timestamp

### 2. floors

Physical floors within a building.

- `floor_id` serial PK
- `building_id` int FK -> buildings.building_id
- `floor_no` int not null
- `created_at` timestamp

Constraint: unique (`building_id`, `floor_no`)

### 3. elevator_shafts

Physical elevator shaft structure within a building (container for elevators).

- `elevator_shaft_id` serial PK
- `building_id` int FK -> buildings.building_id
- `shaft_number` int
- `created_at` timestamp
- `updated_at` timestamp

### 4. elevators

Actual elevator unit housed in a shaft. Each shaft contains one elevator.

- `elevator_id` serial PK
- `elevator_shaft_id` int FK -> elevator_shafts.elevator_shaft_id (1:1 relation)
- `current_floor` int (current floor position)
- `direction` enum('up', 'down', 'idle')
- `capacity` int (max persons)
- `created_at` timestamp
- `updated_at` timestamp

### 5. elevator_status

Real-time operational status log for each elevator (maintenance state tracking).

- `elevator_status_id` serial PK
- `elevator_id` int FK -> elevators.elevator_id
- `status` enum('moving', 'idle', 'maintenance', 'out_of_service')
- `created_at` timestamp
- `updated_at` timestamp

### 6. floorRequest

Floor-level requests generated by passengers (up/down calls from floors).

- `floorRequest_id` serial PK
- `floor_id` int FK -> floors.floor_id (source floor making the request)
- `building_id` int FK -> buildings.building_id
- `requested_at` timestamp
- `status` enum('pending', 'assigned', 'completed')
- `current_floor` int (where passenger is)
- `direction` enum('up', 'down')
- `destination_floor` int (optional target floor)

### 7. ride_assignments

Assignment of an elevator to fulfill a floor request (dispatch decision).

- `ride_assignments_id` serial PK
- `elevator_id` int FK -> elevators.elevator_id
- `floorRequest_id` int FK -> floorRequest.floorRequest_id
- `start_time` timestamp
- `end_time` timestamp
- `floor_visited` int
- `created_at` timestamp
- `updated_at` timestamp

### 8. ride_logs

Detailed historical record of each completed ride (trip execution log).

- `ride_logs_id` serial PK
- `ride_assignments_id` int FK -> ride_assignments.ride_assignments_id
- `elevator_id` int FK -> elevators.elevator_id
- `start_floor` int (pickup floor)
- `end_floor` int (drop floor)
- `start_time` timestamp
- `end_time` timestamp
- `ride_status` enum('ongoing', 'completed', 'pending', 'need_maintenance')
- `created_at` timestamp

### 9. maintenance_check

Maintenance and inspection records for each elevator.

- `maintenance_check_id` serial PK
- `elevator_id` int FK -> elevators.elevator_id
- `issue_type` varchar(100) not null
- `issue_description` text
- `status` enum('pending', 'ongoing', 'completed')
- `scheduled_at` timestamp
- `completed_date` timestamp
- `created_at` timestamp
- `updated_at` timestamp

## Relationship Diagram

```
buildings (1) ──── (N) floors
    │
    ├──── (1) elevator_shafts (1) ──── (1) elevators (1) ──── (N) elevator_status
    │                                           │
    │                                           ├──── (N) ride_assignments ──── (N) ride_logs
    │                                           │
    │                                           └──── (N) maintenance_check
    │
    └──── (N) floorRequest ──── (N) ride_assignments
              │
              └──── (1) floors
```

## Cardinality Details

| Relationship                    | Cardinality | Description                                               |
| ------------------------------- | ----------- | --------------------------------------------------------- |
| buildings → floors              | 1:N         | One building has many floors                              |
| buildings → elevator_shafts     | 1:N         | One building can have multiple elevator shafts            |
| elevator_shafts → elevators     | 1:1         | One shaft contains exactly one elevator                   |
| elevators → elevator_status     | 1:N         | One elevator has multiple status records over time        |
| elevators → ride_assignments    | 1:N         | One elevator is assigned to many requests                 |
| elevators → maintenance_check   | 1:N         | One elevator has many maintenance records                 |
| floors → floorRequest           | 1:N         | One floor generates many requests                         |
| floorRequest → ride_assignments | 1:N         | One request can have multiple assignments (reassignments) |
| ride_assignments → ride_logs    | 1:N         | One assignment can have detailed ride logs                |
| buildings → floorRequest        | 1:N         | One building receives many requests                       |

## Relationship Explanations

### Building Infrastructure Hierarchy

1. **buildings → elevator_shafts → elevators**: Each building contains one or more shafts; each shaft has exactly one elevator unit.
2. **elevators.current_floor** tracks the elevator's current position; **elevators.direction** tracks movement state.
3. **floors** define the vertical levels in each building; they are the source/destination of requests.

### Request and Dispatch Flow

1. **floorRequest** captures passenger requests from a specific floor with a direction.
2. **ride_assignments** represents the dispatch decision: which elevator is sent to handle that request.
3. A single `floorRequest` may be reassigned to different elevators (creating multiple assignment records).

### Real-Time Monitoring

1. **elevator_status** stores snapshots of each elevator's operational state (moving/idle/maintenance/out-of-service).
2. This allows tracking elevator availability and maintenance intervals.

### Ride Execution History

1. **ride_logs** records the actual trip: which floors were visited, timing, and completion status.
2. Each log entry is linked to the original assignment and can reference the elevator for analytics.

### Maintenance Workflow

1. **maintenance_check** documents issues, scheduled maintenance, and completion status.
2. When maintenance is active, the elevator status is updated to 'maintenance' or 'out_of_service'.

## Integrity Rules

- `floorRequest.floor_id` must exist in `floors` table for the same `building_id`.
- `elevators.elevator_shaft_id` must reference a shaft that belongs to `buildings.building_id`.
- `ride_logs.end_time >= ride_logs.start_time` for all completed rides.
- Elevator status transitions follow valid workflow: idle → moving → idle or maintenance → out_of_service → idle.
- Only valid enum values allowed for `direction`, `status`, `ride_status` fields.

## Indexing Plan

- `floors (building_id, floor_no)` unique
- `floorRequest (building_id, status, requested_at desc)` for fast query of pending requests
- `floorRequest (floor_id)` for finding requests from a specific floor
- `ride_assignments (elevator_id, start_time desc)` for elevator dispatch history
- `ride_assignments (floorRequest_id)` for finding assignments linked to a request
- `ride_logs (elevator_id, start_time desc)` for ride history by elevator
- `elevator_status (elevator_id, created_at desc)` for latest status snapshots
- `maintenance_check (elevator_id, status)` for finding active maintenance records
- `elevator_shafts (building_id)` for finding shafts in a building

## Normalization Notes

- Design is normalized to 3NF (Third Normal Form).
- No redundant data; each entity has a clear single responsibility.
- Historical events use append-only log tables: `elevator_status`, `ride_assignments`, `ride_logs`, `maintenance_check`.
- Status tracking is separated from structural data (elevators store structure; elevator_status stores temporal state).
- One-to-one relationship (shaft:elevator) is properly modeled with foreign key constraint.
